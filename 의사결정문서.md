# 의사결정문서 — LLM Wiki 아키텍처 및 설계 결정 기록

각 결정은 **배경 → 선택지 → 결정 및 이유** 형식으로 기록한다.

---

## 결정 1 — Agent를 4개로 분리한 이유

**배경**

초기 설계에서 단일 에이전트가 정리·Q&A·퀴즈·유지보수를 모두 담당하는 구조를 검토했다.

**선택지**

| 방안 | 설명 |
|------|------|
| A — 단일 에이전트 | 하나의 프롬프트가 모든 역할 수행 |
| B — 4개 전용 에이전트 | Edit / Wiki / Quiz / Back 역할 분리 |

**결정: B 채택**

단일 에이전트는 역할 혼동이 심각했다. 프롬프트에 "문제 출제도 하고, Q&A도 하고, 정리도 해라"를 넣으면 Wiki Agent가 자연스럽게 문제를 출제하거나, Quiz Agent가 개념 설명을 시작하는 현상이 반복됐다. 역할 경계를 명시적으로 분리해야만 각 에이전트가 자신의 역할에 집중한다.

- **Edit Agent**: 정리·변환만. 교육자료 외 내용 추가 금지.
- **Wiki Agent**: Q&A만. 문제 출제 요청 시 퀴즈 탭 안내 후 거절.
- **Quiz Agent**: 출제·채점만. 개념 설명 요청 시 위키 탭 안내 후 거절.
- **Back Agent**: 보고만. 발견한 문제를 임의로 수정 금지.

각 에이전트 프롬프트에 `역할 경계 (가장 중요한 규칙)` 섹션을 명시한 것이 이 분리를 강제하는 핵심 장치다.

## 결정 2 — Edit Agent에 `--ingest` 모드를 포함한 이유

**배경**

Edit Agent는 원래 "강의자료를 읽고 정리 노트를 출력"하는 단발 도구였다. 정리된 내용을 wiki에 저장하는 작업은 매번 수동이었다.

**선택지**

| 방안 | 설명 |
|------|------|
| A — 수동 저장 | 에이전트 출력을 사람이 복사해서 wiki에 붙여넣기 |
| B — ingest 모드 내장 | `--ingest` 플래그로 raw/ → wiki/ 파이프라인 자동화 |

**결정: B 채택**

PDF 한 장 정리에 평균 2~3분이 걸리고, 6개 과목 전체 자료가 수십 개라면 수동 저장만으로도 수 시간이 소요된다. 정리 에이전트가 wiki 저장까지 완결하는 구조가 실용적이다.

ingest 모드는 MCP 인터페이스(`tools/wiki_client.py`)를 경유해 wiki에 접근한다. 이미 처리된 파일은 `ingest_status.json`으로 추적해 재처리하지 않으며, Back Agent가 서버 시작 시 미처리 파일을 감지해 자동으로 ingest를 트리거한다. `raw/ → wiki/`까지 사람 개입 없이 완료되는 것이 목표였다.

## 결정 3 — Wiki Agent에 웹검색을 추가한 이유

**배경**

Wiki Agent는 wiki 페이지와 강의 노트만으로 질문에 답변했다. 강의자료에 없는 내용(최신 동향, 구현 예시, 공식 문서 등)을 질문하면 "모른다"고만 답했다.

**선택지**

| 방안 | 설명 |
|------|------|
| A — wiki/강의 노트만 사용 | 컨텍스트 외 질문은 거절 |
| B — 웹검색 fallback 추가 | 1·2순위 컨텍스트 없을 때 DuckDuckGo 검색 |

**결정: B 채택**

학습 도구로서 "모른다"는 답변은 사용자 경험을 크게 떨어뜨린다. 특히 강의자료가 아직 처음부터 ingest되지 않은 초기 단계나, 강의 범위 밖 배경 지식을 묻는 상황에서 웹검색이 유용하다.

DuckDuckGo Search(`duckduckgo-search` 패키지)를 선택한 이유는 **API 키가 불필요**하기 때문이다. 사용자가 별도 계정 없이 즉시 사용 가능하다. 웹검색은 3순위 fallback으로만 동작한다 — wiki와 강의 노트에 답이 있으면 웹검색을 호출하지 않는다.

컨텍스트 우선순위:
1. 현재 열람 중인 강의 노트 (`currentNotes`)
2. wiki 페이지 키워드 검색 결과 (최대 5페이지)
3. DuckDuckGo 웹검색 (1·2순위 컨텍스트 없을 때만)

## 결정 4 — Back Agent를 Python 파일시스템 기반으로 구현한 이유

**배경**

Back Agent는 wiki 상태를 주기적으로 점검하고 log.md에 기록하는 역할이다. 구현 방식으로 외부 서비스(Cron, 데이터베이스, 모니터링 SaaS 등) 활용을 검토했다.

**선택지**

| 방안 | 설명 |
|------|------|
| A — 외부 스케줄러 + DB | Cron + SQLite/PostgreSQL로 상태 관리 |
| B — Python 파일시스템 직접 접근 | `ingest_status.json` + `wiki/log.md` 파일 기반 |

**결정: B 채택**

이 시스템은 로컬에서 단독 실행되는 개인 학습 도구다. 외부 서비스 의존성을 추가하면 설치 부담이 커지고, 네트워크 없이 오프라인으로 사용할 때 Back Agent가 동작하지 않는다.

파일 기반 구조의 장점:
- `ingest_status.json`만 보면 어떤 파일이 처리됐는지 즉시 알 수 있다.
- `wiki/log.md`는 append-only 텍스트 파일로, git에 포함되어 이력이 자동으로 관리된다.
- 백엔드 서버 없이 `python back_agent.py` 단독 실행이 가능하다.
- Back Agent는 발견한 문제를 직접 수정하지 않고 보고만 한다 — 파일시스템 직접 접근 권한을 최소화하는 설계다.

Stop Hook(`--log-check`)은 LLM 호출 없이 파일 기반 점검만 수행한다. 세션 종료마다 LLM을 호출하면 비용/시간이 발생하므로, 빠른 파일 통계만 기록하고 전체 보고서 생성은 명시적 실행 시에만 한다.

## 결정 5 — 손글씨 필기를 무시하기로 한 이유

**배경**

수업 중 교수가 칠판이나 슬라이드에 손으로 쓴 내용, 학생 필기가 일부 PDF에 포함되어 있었다. 이를 처리할지 여부를 결정해야 했다.

**선택지**

| 방안 | 설명 |
|------|------|
| A — 손글씨 OCR 처리 | 이미지에서 텍스트 추출 후 정리 |
| B — 손글씨 무시 | 디지털 텍스트 레이어만 추출 |

**결정: B 채택**

이유는 두 가지다.

첫째, **기술적 한계**: `pypdf`는 디지털 텍스트 레이어만 추출한다. 손글씨는 이미지로 저장되어 있어 추출 자체가 되지 않는다. OCR을 추가하면 `pytesseract`, `tesseract` 바이너리, 언어 팩 등 설치 부담이 크게 늘어나고, 한국어 손글씨 인식 정확도는 낮다.

둘째, **콘텐츠 품질**: 수업 중 손글씨 메모는 불완전한 문장이 많고 맥락 없이 단어만 적혀 있는 경우가 대부분이다. 이를 wiki에 포함하면 노이즈가 된다. 교수 제공 공식 슬라이드만 wiki의 근거 자료로 사용하는 것이 신뢰성이 높다.

대신 **별표(★/☆/*) 표시**는 강의자료 디지털 텍스트에 포함된 기호이므로 추출 가능하다. 별표가 있는 내용은 중요도 높음으로 판단해 더 상세하게 정리하는 방식으로 중요 내용을 선별한다.
