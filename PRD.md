# PRD.md — LLM Wiki 제품 요구사항 및 에이전트 의사결정 저널

이 문서는 LLM Wiki의 목표, 기능 사양, 그리고 구현 과정에서 에이전트와 인간이
함께 수행한 의사결정 라운드를 AgentMEMO Journal 형식으로 기록한다.

---

## Part 1. 제품 요구사항 (PRD)

### 1-1. 배경 및 동기

> "The tedious part of maintaining a knowledge base is not the reading or the thinking — it's the bookkeeping." — Andrej Karpathy

대학교 1학기 기준 학생은 4~6개 전공 과목의 PDF 강의자료를 다루어야 한다.
기존 방식(파일 탐색, 텍스트 검색)은 "이 개념이 어느 자료에 있었는지" 찾는 데 과도한 시간을 소비한다.
시험 기간에는 정리된 자료가 없어 처음부터 다시 읽어야 한다.

LLM Wiki는 이 **북키핑 비용을 AI 에이전트에게 위임**하여, 학생이 읽기와 사고에만 집중할 수 있게 한다.

### 1-2. 목표 (Goals)

| 번호 | 목표 | 성공 지표 |
|------|------|-----------|
| G1 | 강의자료를 자연어로 검색·질의 가능하게 함 | 파일명 몰라도 개념으로 답변 받기 |
| G2 | 시험 준비를 자동화 (문제 출제·채점) | 파일 선택 후 문제 생성까지 1분 이내 |
| G3 | 강의자료 변경 시 자동 반영 | 새 파일 추가 후 서버 재시작만으로 반영 |
| G4 | 학생이 설치·운영하기 쉬운 로컬 서비스 | `bash start.sh` 한 줄로 실행 |

### 1-3. 비목표 (Non-Goals)

- 다중 사용자 지원 (단일 학생의 로컬 도구)
- 실시간 협업 편집
- 강의자료 무단 배포 (raw/ 폴더는 로컬에만 존재)
- 범용 지식 검색 (구글 대체 아님)

### 1-4. 핵심 기능 사양

#### F1. Edit Agent — 강의자료 자동 정리

- **입력**: PDF 파일 경로 (raw/{과목}/파일.pdf)
- **처리**: pypdf로 텍스트 추출 → Claude에 정리 프롬프트 전달
- **출력**: 마크다운 정리 노트 (스트리밍)
- **규칙**: 별표 내용 강조, 손글씨 무시, 시험 제외 내용 필터
- **부작용**: ChromaDB에 청크 임베딩, wiki 페이지 자동 생성

#### F2. Wiki Agent — 자연어 Q&A

- **입력**: 자연어 질문
- **컨텍스트 구성**: 현재 열람 강의노트 + wiki 키워드 검색 + 웹 검색 fallback
- **출력**: 마크다운 답변 + 출처 명시 (스트리밍)
- **역할 경계**: 개념 설명만, 문제 출제 거절

#### F3. Quiz Agent — 문제 출제 및 채점

- **문제 생성 입력**: 정리 노트 텍스트, 문제 수
- **문제 생성 출력**: 객관식 문제 + QUIZ_ANSWER/QUIZ_EXPLAIN 마커 (정답 숨김)
- **채점 입력**: 문제 텍스트, 학생 답변
- **채점 출력**: 정오 판정 + 오답 노트
- **역할 경계**: 퀴즈/채점만, 개념 질문 거절

#### F4. Back Agent — 자동 유지보수

- **트리거**: 서버 시작 시 자동 실행
- **점검 항목**: 미처리 raw 파일, 오래된 wiki 페이지, 깨진 링크
- **출력**: log.md에 보고서 append
- **자동화**: 미처리 파일 발견 시 edit_agent --ingest 자동 호출

#### F5. 웹 UI — 단일 HTML 파일

- **레이아웃**: 3패널 (네비게이션 / 본문 / 에이전트)
- **네비게이션**: 과목 폴더 트리, 파일명 검색
- **본문**: 마크다운 렌더링, Edit Agent 스트리밍
- **에이전트 패널**: 위키 탭(채팅), 퀴즈 탭(출제·채점)
- **테마**: 다크 테마 (Tokyo Night 계열), 가독성 우선 색상

### [ROUND-01] RESEARCH | Wiki 도구 아키텍처 탐색

**날짜:** 2026-05-26  
**상태:** CLOSED

**질문:** 강의자료 기반 개인 wiki를 어떻게 구조화할 것인가?

**에이전트 분석:**
- Karpathy의 "북키핑 비용 위임" 원칙에서 출발
- 단순 RAG(검색→답변)만으로는 지식이 누적되지 않음
- wiki 페이지를 LLM이 직접 생성하고 cross-reference하는 구조 제안

**결정:**
```
raw/  →  Edit Agent  →  wiki/pages/  →  Wiki Agent  →  학생
(원본)    (정리·합성)    (영구 저장)     (질의 응답)
```

**근거:** RAG는 검색 시마다 같은 계산을 반복하지만, 사전 합성된 wiki 페이지는
쿼리마다 축적된 이해를 제공한다. "인덱스 시간에 사고, 쿼리 시간에 검색"
